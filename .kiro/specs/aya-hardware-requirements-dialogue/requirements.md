# 要件仕様書

## 概要

本機能は、ユーザーがAYA（AIアシスタント）との対話を通じて、組み込みシステムの要件定義を段階的に詳細化し、最終的にハードウェアシステムの提案を受けるまでのプロセスを実現します。今回の実装範囲は、全体フローのステップ1〜3（要望入力、要件定義作成、対話による詳細化）に限定されます。

## 全体システムフロー（参考）

1. **ユーザーがチャットから製作したい組み込みシステムの要望を伝える** ← 今回実装
2. **AYAがユーザーの要望をもとにシステムの要件定義を作成する** ← 今回実装
3. **ユーザーに質問を行いながらシステムの要件定義をより鮮明にする** ← 今回実装
4. ユーザーが要件定義を承認すると要件にあったハードウェアシステム提案を行う
5. AYAは部品の互換性を自動で確認
6. 価格と納期をもとに部品を選定
7. 設計を承認すると発注リストが作成
8. 部品到着後の組み立てサポート
9. エラー対応とログ記録

## 要件（実装範囲：ステップ1〜3）

**注記**: 要件7〜9は既存システムとの統合要件として追加されました。

### 要件1: チャットインターフェースでの要望入力

**ユーザーストーリー:** 組み込みエンジニアとして、自然な会話形式で製作したいシステムの要望をAYAに伝えたい。これにより、技術的な仕様書を最初から作成する必要なく、アイデアから始められる。

#### 受け入れ基準

1. WHEN ユーザーがチャットに「温度センサーを使った監視システムを作りたい」と入力する THEN AYAが要望を受け付け、初期分析を開始する
2. WHEN ユーザーが曖昧な表現（「小型のIoTデバイス」等）を使用する THEN AYAが理解した内容を確認メッセージで返す
3. IF 複数の解釈が可能な要望の場合 THEN AYAが選択肢を提示して確認する
4. WHEN 音声入力が使用される THEN テキストに変換されてチャットに表示される

### 要件2: AIによる要件定義の自動生成

**ユーザーストーリー:** 組み込みエンジニアとして、入力した要望からAYAが自動的に要件定義の初版を作成してほしい。これにより、ゼロから要件を考える時間を削減できる。

#### 受け入れ基準

1. WHEN ユーザーの要望が入力される THEN AYAが30秒以内に要件定義の初版を生成する
2. WHEN 要件定義が生成される THEN 以下の項目が含まれる：
   - システムの目的と概要
   - 機能要件（主要機能のリスト）
   - 非機能要件（性能、信頼性、環境条件等）
   - 制約条件（サイズ、コスト、電源等）
3. IF 類似プロジェクトの事例がある THEN 参考情報として提示される
4. WHEN 要件定義が表示される THEN ユーザーが編集可能な形式で提供される

### 要件3: 対話による要件の詳細化と質問生成

**ユーザーストーリー:** 組み込みエンジニアとして、AYAとの対話を通じて要件を段階的に詳細化したい。AYAが適切な質問をしてくれることで、見落としがちな要件も明確にできる。

#### 受け入れ基準

1. WHEN 初期要件定義が生成される THEN AYAが不足情報を特定し、優先度順に質問リストを生成する
2. WHEN AYAが質問する際 THEN 以下の形式で提示される：
   - 質問の意図（なぜこの情報が必要か）
   - 具体的な質問内容
   - 回答例または選択肢（該当する場合）
3. IF ユーザーが「わからない」と回答した場合 THEN AYAが一般的な選択肢や業界標準を提案する
4. WHEN ユーザーが回答する THEN AYAが要件定義をリアルタイムで更新し、変更箇所をハイライト表示する

### 要件4: 要件定義の構造化と可視化

**ユーザーストーリー:** 組み込みエンジニアとして、対話を通じて作成された要件定義を構造化された形式で確認したい。これにより、抜け漏れがないか視覚的に把握できる。

#### 受け入れ基準

1. WHEN 要件定義が更新される THEN 以下の構造で整理される：
   - システム要件（全体像）
   - ハードウェア要件（CPU、メモリ、I/O等）
   - ソフトウェア要件（OS、ミドルウェア、アプリケーション）
   - インターフェース要件（通信、電源、機械的接続）
2. WHEN 要件間の依存関係がある THEN 視覚的に関連性が表示される
3. IF 要件に矛盾や競合がある THEN 警告アイコンと説明が表示される
4. WHEN 進捗状況を確認する THEN 各カテゴリの完成度がパーセンテージで表示される

### 要件5: コンテキストの保持と対話履歴管理

**ユーザーストーリー:** 組み込みエンジニアとして、長時間の対話や複数セッションにわたる要件定義作業でも、コンテキストが保持されることを期待する。これにより、効率的に作業を継続できる。

#### 受け入れ基準

1. WHEN ユーザーが前回の内容を参照する THEN AYAが過去の対話内容を正確に把握して応答する
2. WHEN セッションが中断される THEN 自動保存され、次回ログイン時に継続可能
3. IF 矛盾する要求が後から追加される THEN AYAが過去の決定事項との矛盾を指摘する
4. WHEN 対話履歴を検索する THEN キーワードや日時で過去のやり取りを検索できる

### 要件6: 要件定義のレビューと承認プロセス

**ユーザーストーリー:** 組み込みエンジニアとして、完成した要件定義をレビューし、承認することで次のステップ（ハードウェア提案）に進みたい。これにより、要件の品質を確保し、適切なシステム提案を受けられる。

#### 受け入れ基準

1. WHEN 要件定義が一定の完成度に達する THEN AYAがレビューを提案する
2. WHEN レビューモードに入る THEN 以下の項目がチェックされる：
   - 必須項目の記入状況
   - 数値の妥当性（電力、サイズ等）
   - 実現可能性の初期評価
3. IF 改善提案がある THEN AYAが具体的な修正案を提示する
4. WHEN レビューが完了する THEN ユーザーが要件定義を承認できる
5. WHEN 要件定義が承認される THEN 承認ステータスが「承認済み」に更新され、システム提案が可能になる
6. IF 要件定義が未承認の場合 THEN システム提案を要求しても要件定義の承認を促すメッセージが表示される

### 要件7: Auto Devlogとの統合による開発ドキュメント管理

**ユーザーストーリー:** 組み込みエンジニアとして、AYAとの対話で生成された要件定義や設計情報をAuto Devlogタブで一元管理したい。これにより、プロジェクトの全ドキュメントを効率的に管理・参照できる。

#### 受け入れ基準（段階的実装）

**フェーズ1（実装済み）:**

1. WHEN AIが外部リソースを参照する THEN AI参考文献がAuto Devlogに自動記録される
2. WHEN Auto Devlogタブを開く THEN AI参考文献の一覧が表示される
3. IF フィルターを使用する THEN ソース（公式/GitHub等）やカテゴリ（仕様/互換性/実装）で絞り込み可能

**フェーズ2（今回実装）:**

1. WHEN 要件定義が生成される THEN 自動的にAuto Devlogに保存される
2. WHEN 要件定義が保存される THEN 要件6で定義された承認ステータスがAuto Devlogでも参照可能になる
3. WHEN 対話で重要な決定がなされる THEN 「決定事項」として自動記録される
4. IF ユーザーが手動で追加したメモがある THEN Auto Devlogに統合される

**フェーズ3（将来実装）:**

1. IF ユーザーがドキュメントを追加・編集する THEN リッチテキストエディタ（要件10参照）で編集可能
2. WHEN ドキュメントを検索する THEN 全文検索とタグベースのフィルタリングが可能
3. WHEN エクスポートを要求する THEN PDF/Markdown/Word形式で出力可能
4. IF ユーザーが手動でメモを追加する THEN プロジェクトメモとして保存される
5. WHEN 画像を挿入する THEN ドラッグ&ドロップまたはペーストで簡単に追加可能

### 要件8: 既存チャットシステムとの深い統合

**ユーザーストーリー:** 組み込みエンジニアとして、既存のChatPanelとAuto Devlogがシームレスに連携し、対話から自動的にドキュメントが生成・整理されることを期待する。これにより、手動でのドキュメント作成作業を最小化できる。

#### 受け入れ基準

1. WHEN ChatPanelで要件定義モードが開始される THEN 専用のインテント（requirements_definition）が追加される
2. WHEN AYAが要件定義を生成する THEN ChatPanelLogicが自動的にAuto Devlogへ保存する
3. WHEN システム提案（suggest_system）インテントが検出される THEN 要件9で定義されたプロセスに従って処理される
4. IF チャット内で「要件をドキュメント化」と入力する THEN 現在の対話内容から要件定義ドキュメントが生成される
5. WHEN 既存のintentClassifierが動作する THEN requirements_definitionインテントが適切に分類される
6. IF ユーザーがチャット内で「@devlog」タグを使用する THEN その内容が直接Auto Devlogに保存される

### 要件9: 要件定義書ベースのシステム提案

**ユーザーストーリー:** 組み込みエンジニアとして、AYAがシステム提案を行う際は、必ず承認済みの要件定義書を基にして欲しい。これにより、要件に沿った適切なシステム構成が提案される。

#### 受け入れ基準

1. WHEN ユーザーがシステム提案を要求する THEN AYAは承認済み要件定義書の存在を確認する
2. IF 承認済み要件定義書が存在しない場合 THEN 以下の処理を実行する：
   - 「要件定義書が未作成または未承認です」というメッセージを表示
   - 要件定義の作成を開始するか確認する
   - ユーザーが同意した場合、要件定義モードに移行する
3. IF 承認済み要件定義書が1つ存在する場合 THEN その要件定義書を基にシステム提案を生成する
4. IF 承認済み要件定義書が複数存在する場合 THEN 以下の処理を実行する：
   - 利用可能な要件定義書のリストを表示
   - ユーザーにどの要件定義書を基にするか選択を求める
   - 選択された要件定義書を基にシステム提案を生成する
5. WHEN システム提案が生成される THEN 以下の情報が含まれる：
   - 参照した要件定義書の名称とバージョン
   - 要件との対応関係（どの要件に対してどのコンポーネントが提案されているか）
   - 要件を満たすための追加推奨事項
6. IF 要件定義書に曖昧な点がある場合 THEN AYAは明確化のための質問を行う

### 要件10: リッチテキストエディタによるドキュメント編集

**ユーザーストーリー:** 組み込みエンジニアとして、要件定義書やドキュメントをスムーズに編集したい。画像の挿入、バージョン管理などの機能により、効率的なドキュメント作成が可能になる。

#### 受け入れ基準

1. WHEN ドキュメントを編集する THEN 以下の機能が利用可能：
   - テキストフォーマット（太字、斜体、下線、取り消し線、コード）
   - 構造化要素（見出し、段落、リスト、表、引用）
   - 画像・図表の挿入（ドラッグ&ドロップ、クリップボード貼り付け対応）
   - リンクの挿入と管理
   - Mermaidダイアグラムの埋め込み
2. WHEN コメントを追加する THEN 選択テキストに対してスレッド形式でコメント可能
3. WHEN 編集を行う THEN 3秒間の無操作後に自動保存される
4. WHEN バージョン履歴を確認する THEN 過去のバージョンとの差分表示と復元が可能
5. IF 画像をアップロードする THEN 自動的に圧縮・最適化される
6. WHEN ドキュメントをエクスポートする THEN PDF、Word、Markdown形式での出力が可能
7. IF Google Docsからコピー&ペーストする THEN フォーマットが可能な限り保持される
